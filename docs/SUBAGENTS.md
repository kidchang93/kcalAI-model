## 병렬화 기준

### 병렬로 안전한 작업

- 서로 다른 도메인 파일 읽기 (`api/auth_api.py` vs `main.py`)
- 문서 초안 작성 (파일이 겹치지 않을 때)
- 라우트 전수 조사, 환경변수 사용처 추적, 의존성 방향 검사 등 **읽기 전용 조사**
- 앱 저장소(`k-calAI-RN`)와 서버 저장소를 각각 조사

### 반드시 순차로 해야 하는 작업

| 작업 | 이유 |
|------|------|
| `main.py` 수정 | 라우터 등록·CORS·모델 로딩이 한 파일에 모여 있어 충돌합니다 |
| `api/__init__.py` 수정 | 라우터 재수출 지점. 동시 편집 시 덮어씁니다 |
| `database.py`의 `init_db()` import 목록 수정 | 같은 함수를 여러 에이전트가 건드립니다 |
| 새 도메인 추가 (schemas → services → api → main) | 뒤 단계가 앞 단계의 시그니처에 의존합니다 |
| DB 스키마 변경 | `create_all` 특성상 순서와 볼륨 상태에 의존합니다 |
| `requirements.txt` 수정 | 단일 파일. 두 에이전트가 쓰면 충돌합니다 |

**파일 단위로 소유권을 나눌 수 없으면 병렬화하지 않습니다.** 이 저장소는 도메인당 파일이 하나씩이라 병렬 구현의 이득이 작습니다.

### worktree 격리가 필요한 경우

여러 에이전트가 동시에 파일을 쓰는 구현 작업에만 `isolation: "worktree"`를 씁니다. 다만 이 저장소는 `venv/`, `__pycache__/`, 모델 캐시가 worktree에 복제되지 않으므로 **worktree에서 서버를 기동해 검증할 수 없습니다.** 검증은 메인 워크트리로 병합한 뒤 수행합니다.

## 결과물 검수

서브에이전트 결과를 반영하기 전에 다음을 직접 확인합니다. 에이전트의 "완료했습니다" 보고를 그대로 신뢰하지 않습니다.

1. `docs/CODE_STYLE.md`의 **금지 패턴 표**에 걸리는 코드가 없는가.
2. `docs/REVIEW.md`의 **레이어 / 보안 / API 계약** 체크리스트를 통과하는가.
3. 새 환경변수가 `.env.example`에 추가되었는가.
4. API 계약이 바뀌었다면 `k-calAI-RN/services/*.ts`와 경로·필드가 일치하는가.
5. `uvicorn main:app`이 기동하고 `/docs`가 200을 반환하는가. **(에이전트가 아니라 사람이 확인)**

## 에이전트에게 시키지 말 것

- **`/api/gpt-predict` 호출.** HF Inference API를 실제로 태우므로 토큰 사용량이 발생합니다.
- **`git push`.** 특히 `dev` 브랜치는 push 즉시 NCP 서버로 배포됩니다.
- **배포 워크플로 수정.** `deploy.yml`은 시크릿·서버 상태에 의존하므로 사용자 확인 없이 건드리지 않습니다.
- **`runs/` 아래 파일 추가·삭제.** 70MB 학습 산출물입니다. 스토리지 정책은 제품 결정입니다.
- **`.env` 수정.** `HF_TOKEN`과 NCP 자격증명이 들어 있습니다.

## 두 저장소를 함께 다루는 세션

`kcalAI-model`과 `k-calAI-RN`은 하나의 기능 체인입니다. API 계약을 건드리는 세션에서는 두 저장소를 함께 엽니다.

```
Explore  →  서버 라우트 전수 조사        (kcalAI-model)
Explore  →  앱이 호출하는 URL 전수 조사  (k-calAI-RN)     ← 병렬 가능
         ↓
        diff 확인 후 사용자에게 계약 확정 요청
         ↓
general-purpose → 서버 수정   ┐
general-purpose → 앱 수정     ┘  ← 계약 확정 후에만, 순차 또는 파일 소유권 분리
```

계약이 확정되기 전에 구현 에이전트를 띄우지 않습니다.
