# kCalAIModel Project Conventions

## 현재 기술 스택

- FastAPI 기반 Python 서버
- `main.py`에서 앱 생성 및 라우터 등록
- `api`, `services`, `schemas`로 역할 분리

## 세션 운영 규칙

- 이 저장소를 수정할 때 앱 소비 방식이 바뀌면 `kcalai`를 같은 세션에서 함께 확인한다.
- 엔드포인트, 요청 필드, 응답 모델 변경은 프런트엔드 영향 분석 없이 머지하지 않는다.
- 모델 성능 작업과 제품 API 안정화 작업을 같은 변경으로 섞지 않는 것을 원칙으로 한다.

## 구조 규칙

- `api`는 HTTP 입출력과 예외 변환에 집중한다.
- `services`는 추론, 파일 업로드, 외부 연동 같은 비즈니스 로직을 담당한다.
- `schemas`는 요청/응답 계약의 단일 기준으로 유지한다.
- 실험용 자산, 모델 파일, 로그는 제품 코드와 경계를 명확히 둔다.

## API 설계 규칙

- 앱이 의존하는 공개 계약은 명시적인 스키마로 유지한다.
- 성공 응답과 실패 응답 형태를 일관되게 유지한다.
- 라우트 경로는 앱 코드와 반드시 맞아야 하며, 변경 시 두 저장소를 함께 수정한다.
- 모델 내부 클래스명이나 라이브러리 세부사항을 API에 직접 노출하지 않는다.

## 모델 및 서비스 규칙

- 모델 로딩 위치와 방식은 서버 시작 시간, 메모리 사용량, 요청 지연을 고려해 관리한다.
- 추론 결과는 제품에서 바로 쓰기 쉬운 필드 중심으로 가공해서 반환한다.
- 추론 정확도 실험 코드는 서비스 코드와 분리하고, 실험 결과를 API 계약 변경의 근거로 삼지 않는다.

## 운영 및 품질 기준

- 로컬 개발 시 CORS, 업로드, 예외 로그를 먼저 점검한다.
- 새 엔드포인트 추가 시 최소한 요청 예시와 응답 예시를 문서에 남긴다.
- 서버 내부 리팩터링이라도 앱 계약에 영향이 있으면 breaking change로 취급한다.
